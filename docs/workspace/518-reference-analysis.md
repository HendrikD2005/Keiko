# Epic #518 — Open-Source Workspace Reference Analysis

Status: Wave 1 deliverable for [issue #520](https://github.com/oscharko-dev/Keiko/issues/520) under parent epic [#518](https://github.com/oscharko-dev/Keiko/issues/518).

Audit date: 2026-06-06. Companion to [518-capability-audit.md](518-capability-audit.md).

## Purpose

This document studies four public open-source projects that already implement infinite-canvas, document-workspace, whiteboard, and node-based workflow patterns, then maps each useful concept to a Keiko-specific decision. The output is documentation only. No source code from any reference project may be copied; no new third-party dependency may be proposed.

Every concept maps to one of: **adopt concept**, **adapt concept**, **reject**, **defer**, or **requires ADR**.

## How Keiko already satisfies most reference concepts

The capability audit confirmed that Keiko's `packages/keiko-ui/src/app/components/desktop/` already implements an editor-model workspace shell with world coordinates, camera, viewport, window/object registry, command palette, connections, and an extension contract. The reference analysis below is therefore a _mapping_ exercise, not a _proposal_ exercise. Concepts already present in Keiko receive **adapt concept (existing surface)** or **reject (already covered)** dispositions.

## Reference project 1 — tldraw

### Architecture concepts covered

| tldraw concept               | Public behavior summary                                                                                                                                                                                      | Keiko mapping                                                                                                                                                                            | Disposition                                                                                                                                                                |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Editor`                     | Single source-of-truth class that owns the document, camera, selection, hover, focus, undo/redo, and commands. Components subscribe to editor state.                                                         | `useWorkspace` is Keiko's editor-equivalent: owns windows, camera (`zoom/x/y`), focus, connections, and the workspace action surface.                                                    | adapt concept (existing surface) — formalize `useWorkspace` as Keiko's "workspace editor" in #525 ADR                                                                      |
| `Camera` / `Viewport`        | A camera has `(x, y, z)` (z = zoom). The viewport applies camera transforms to world coordinates. tldraw separates camera state from viewport DOM size and offers `setCamera`, `centerOnPoint`, `zoomToFit`. | Keiko's `View { zoom, x, y }` plus `panBy` and zoom helpers are functionally identical. Fit-to-view is implementable on top of the current surface.                                      | adapt concept (existing surface) — record `View` as the camera contract; add fit-to-view helper inside `useWorkspace` only if a downstream issue actually needs it         |
| `Shape` / `ShapeUtil`        | A shape is a typed record `{ id, type, x, y, props }`; a `ShapeUtil` declares behavior (render, hit, resize, indicator, geometry). New shape types register via `defineShape`.                               | Keiko's `AppWindow { id, type, x, y, w, h, z, cfg }` mirrors the shape record; `WindowTypeDef` mirrors `ShapeUtil`; `registerWindowRender` mirrors `defineShape`.                        | adopt concept — confirm the parallel in #525 ADR; extend `WindowTypeDef` with the `lifecycle`/`trustBoundary`/`authority`/`persistence` fields from the audit's Gap Matrix |
| `Tool`                       | Tool state machine for the active interaction mode (select, draw, hand, etc.). Tools own pointer state, cursor, and gestures.                                                                                | Keiko's interactions are command-driven (palette + buttons) rather than mode-driven. The command palette + window-level interactions cover the same need without a state-machine.        | adapt concept (rejected as a state machine, adopted as commands) — record in #523 UX blueprint                                                                             |
| `Store` / `Records`          | Reactive document store of typed records with serializable schema. tldraw uses it for persistence and sync.                                                                                                  | Keiko's UI state lives in React hooks (`useWorkspace`, contexts). Durable layout uses `node:sqlite` (#62). Evidence and workspace FS have their own stores.                              | adapt concept (rejected as a single store) — keep multi-store ownership per existing package boundaries                                                                    |
| Undo/redo                    | tldraw exposes a `history` API and command pattern. Most editor actions are undoable; programmatic actions are explicitly excluded.                                                                          | Not implemented today. Per audit Gap Matrix, a typed undo stack lands in #527, scoped to UI-state actions and refusing to mutate evidence / patches / verification / model-call records. | adapt concept — fully specified in #523 UX blueprint and #525 ADR; implemented in #527                                                                                     |
| Collaboration boundary       | tldraw v3 added multiplayer via `useSyncStore` and a sync server.                                                                                                                                            | Real-time collaboration is explicitly out of scope for #518. WebSocket already in use for streaming; WebRTC is deferred.                                                                 | reject (explicit non-goal)                                                                                                                                                 |
| Virtualization / performance | tldraw culls off-screen shapes, batches paints, and uses `requestAnimationFrame` for camera animation.                                                                                                       | Keiko's window count is small (≤dozens); culling is unnecessary at present scale. Existing `useWorkspace` already minimizes re-renders by memoizing window arrays.                       | defer — record as future ADR candidate if window counts grow                                                                                                               |
| React/TypeScript integration | tldraw exposes typed hooks (`useEditor`, `useValue`, `useUiContext`).                                                                                                                                        | Keiko's `useWorkspace`/`useChatSession`/`useTwinMode`/`useTheme` hooks already form a typed integration layer.                                                                           | adapt concept (existing surface)                                                                                                                                           |

### Concepts explicitly rejected

- **`@tldraw/tldraw` runtime dependency.** Forbidden by the no-new-dependency invariant. Substrate already exists in Keiko.
- **tldraw `defineMigrations` schema-versioning system.** Keiko already has `evidenceSchemaVersion` and `node:sqlite` `user_version` patterns; no second migration system is needed.
- **tldraw "asset" subsystem.** Out of scope for #518.

## Reference project 2 — AFFiNE

### Architecture concepts covered

| AFFiNE concept                    | Public behavior summary                                                                        | Keiko mapping                                                                                                                                                                                                                                     | Disposition                                                                                 |
| --------------------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Document + whiteboard composition | AFFiNE composes a document mode and an edgeless (whiteboard) mode that share a block model.    | Keiko already composes a desktop workspace with multiple object types (chat, files, editor, browser, terminal, review, agents, integ, project, plus panels). The "document/whiteboard" duality already exists as `WindowType` plus `tool` panels. | adapt concept (existing surface)                                                            |
| Workspace structure               | AFFiNE workspaces own collections of documents and a sidebar.                                  | Keiko's `LeftRail` plus `Header` workspace tabs already provide the navigation. Project-scoped chats and capsules already exist.                                                                                                                  | adapt concept (existing surface) — extend Header tabs only if #523 UX blueprint requires it |
| BlockSuite block model            | Each document is a tree of typed blocks (paragraph, list, code, image, embed, edgeless frame). | Keiko does not need a fine-grained block model for the workspace foundation. Existing `SafeMarkdown` renders rich text; existing widgets render typed cards. The unit of composition is the window/widget, not the block.                         | reject (out of scope) — re-evaluate if a future "Document object" issue requires it         |
| Local-first assumptions           | AFFiNE stores documents locally and syncs optionally.                                          | Keiko is local-first by design: `node:sqlite` UI persistence, FS-backed workspace, evidence store.                                                                                                                                                | adopt concept — already in place                                                            |
| Knowledge base / linked notes     | AFFiNE exposes bidirectional links and a knowledge graph.                                      | Keiko's local-knowledge package already implements a capsule graph with a connector graph UI.                                                                                                                                                     | adapt concept (existing surface)                                                            |
| Persistence / sync boundary       | AFFiNE separates local store from optional Cloud sync.                                         | Keiko has no Cloud sync; evidence + workspace + memory stores are local.                                                                                                                                                                          | reject (explicit non-goal) — record local-only constraint in #525 ADR                       |
| Multi-workspace switching         | AFFiNE supports multiple top-level workspaces.                                                 | Keiko's project sidebar (per memory) already supports multiple projects, scoped chats, and project switching.                                                                                                                                     | adapt concept (existing surface)                                                            |
| Knowledge-work workflows          | AFFiNE has templates, AI commands, and structured property pages.                              | Keiko has workflow descriptors, the run/evidence ledger, and grounded chat.                                                                                                                                                                       | adapt concept (existing surface)                                                            |

### Concepts explicitly rejected

- **`@blocksuite/*` runtime dependencies.** Forbidden by the no-new-dependency invariant.
- **AFFiNE Cloud sync.** Out of scope; Keiko is local-first.
- **AFFiNE plugin host.** Reuse `registerWindowRender` if extension is needed; no runtime plugin host beyond that.

## Reference project 3 — Excalidraw

### Architecture concepts covered

| Excalidraw concept | Public behavior summary                                                                                           | Keiko mapping                                                                                                                                                                                              | Disposition                                                                               |
| ------------------ | ----------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Scene model        | A flat array of `ExcalidrawElement` records; rendering iterates the array.                                        | Keiko's `AppWindow[]` is functionally identical.                                                                                                                                                           | adapt concept (existing surface)                                                          |
| Coordinate system  | World coordinates with a `(scrollX, scrollY, zoom)` viewport. Pointer events are converted via affine transforms. | Keiko's `View { zoom, x, y }` plus `Workspace.tsx` pointer handlers already implement this.                                                                                                                | adapt concept (existing surface)                                                          |
| Renderer           | Excalidraw renders to a 2D canvas. Each frame redraws affected layers.                                            | Keiko renders windows as DOM elements with CSS transforms (the workspace background is a `WorkspaceShader` div). DOM rendering is sufficient at Keiko's element count and provides accessibility for free. | adapt concept (rejected as canvas) — explicit DOM rendering decision recorded in #525 ADR |
| Hit detection      | Excalidraw uses bounding-box hit tests plus geometric intersection helpers.                                       | Keiko uses DOM hit-testing for windows (browser handles it) and explicit hit logic in `connectionUtils.ts` for connections.                                                                                | adapt concept (existing surface)                                                          |
| Selection logic    | Excalidraw supports single, multi, and marquee selection.                                                         | Keiko's selection is single-window today; multi-selection is a candidate extension for #527 if #523 UX requires it.                                                                                        | adapt concept (extension in #527)                                                         |
| Command system     | Excalidraw commands are dispatched as actions through a reducer.                                                  | Keiko commands are typed records routed through `CommandPalette` and the `WorkspaceApi`.                                                                                                                   | adapt concept (existing surface) — no reducer is needed                                   |
| Collaboration      | Excalidraw supports collaboration via Firebase.                                                                   | Out of scope for #518.                                                                                                                                                                                     | reject                                                                                    |

### Concepts explicitly rejected

- **`@excalidraw/excalidraw` runtime dependency.** Forbidden by the no-new-dependency invariant.
- **Roughjs / hand-drawn render style.** Not a Keiko design direction.
- **`React-Konva` or `canvas` library adoption.** DOM rendering is sufficient.

- **`@xyflow/react` runtime dependency.** Forbidden by the no-new-dependency invariant.

## No-new-dependency compliance note

Every adopted/adapted concept in this analysis is implementable on Keiko's existing stack and platform APIs:

| Capability            | Implementation seam in Keiko                                                             | New dependency?                 |
| --------------------- | ---------------------------------------------------------------------------------------- | ------------------------------- |
| Editor pattern        | React hooks (`useWorkspace`) + TypeScript types                                          | No                              |
| Camera transforms     | Pure math (`zoom`, `x`, `y`) + CSS `transform` on the workspace surface                  | No                              |
| Shape/object registry | TypeScript discriminated union + a single registry map                                   | No                              |
| Undo stack            | Pure data structure with typed `Action` records                                          | No                              |
| DOM rendering         | React component tree + CSS                                                               | No                              |
| Node + edge rendering | Existing SVG/CSS patterns in `ConnectionsLayer` and `connector-graph`                    | No                              |
| Persistence           | `node:sqlite` (already in use via #62)                                                   | No                              |
| Pointer behavior      | `pointerdown`/`pointermove`/`pointerup` + `PointerEvent` (browser native)                | No                              |
| Keyboard handling     | `keydown` listeners + Mousetrap-style keymap object (pure data)                          | No                              |
| Accessibility         | Existing `jest-axe`, axe-core, aria patterns, focus-visible                              | No                              |
| WebSocket streaming   | Existing `ws` already in the product architecture                                        | No                              |
| Future WebRTC         | Not implemented; if a future ADR approves it, browser native `RTCPeerConnection` is used | No (browser native if approved) |

The capability audit's Gap Matrix produces zero new runtime dependencies. This reference analysis confirms that constraint at the concept level: no concept selected for adoption requires a third-party library.

## Follow-up ADR candidates

The analysis identifies the following ADRs to be written by #525:

1. ADR — Keiko workspace substrate: `useWorkspace` is the editor; DOM rendering is the renderer; the existing `View`/`Workspace.tsx`/`useWorkspace` define camera and viewport.
2. ADR — Object registry contract: extend `WindowTypeDef` with `lifecycle`, `trustBoundary`, `authority`, `persistence` fields; registration-time validator.
3. ADR — Undo/redo boundary: typed action records; UI-only mutations are reversible; evidence/patch/verification/model-call mutations are refused by type at compile time.
4. ADR — State ownership and persistence: per-package stores (`node:sqlite` for UI durable, `keiko-evidence` for evidence, `keiko-workspace` for FS, `keiko-memory-vault` for memory) are explicitly documented as the workspace's persistence surface; no new store is added.
5. ADR — Security and evidence flows in the workspace foundation: all model calls remain behind the Model Gateway; all tool calls remain inside `keiko-tools` terminal-policy; all evidence remains redacted-by-construction; the workspace shell carries no new credential surface.
