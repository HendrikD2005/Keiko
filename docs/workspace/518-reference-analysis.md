# Epic #518 ŌĆö Open-Source Workspace Reference Analysis

Status: Wave 1 deliverable for [issue #520](https://github.com/oscharko-dev/Keiko/issues/520) under parent epic [#518](https://github.com/oscharko-dev/Keiko/issues/518).

Audit date: 2026-06-06. Companion to [518-capability-audit.md](518-capability-audit.md).

## Purpose

This document studies four public open-source projects that already implement infinite-canvas, document-workspace, whiteboard, and node-based workflow patterns, then maps each useful concept to a Keiko-specific decision. The output is documentation only. No source code from any reference project may be copied; no new third-party dependency may be proposed.

Every concept maps to one of: **adopt concept**, **adapt concept**, **reject**, **defer**, or **requires ADR**.

## How Keiko already satisfies most reference concepts

The capability audit confirmed that Keiko's `packages/keiko-ui/src/app/components/desktop/` already implements an editor-model workspace shell with world coordinates, camera, viewport, window/object registry, command palette, connections, and an extension contract. The reference analysis below is therefore a _mapping_ exercise, not a _proposal_ exercise. Concepts already present in Keiko receive **adapt concept (existing surface)** or **reject (already covered)** dispositions.

## Reference project 1 ŌĆö tldraw

### Product, interaction, and architecture concepts covered

| tldraw concept               | Concept class             | Public behavior summary                                                                                                                                                                                      | Keiko mapping                                                                                                                                                                                                   | Disposition                                                                                                                                                        |
| ---------------------------- | ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `Editor`                     | architecture              | Single source-of-truth class that owns the document, camera, selection, hover, focus, undo/redo, and commands. Components subscribe to editor state.                                                         | `useWorkspace` is Keiko's editor-equivalent: owns windows, camera (`zoom/x/y`), focus, connections, and the workspace action surface.                                                                           | adapt concept (existing surface) ŌĆö formalize `useWorkspace` as Keiko's "workspace editor" in #525 ADR                                                              |
| `Camera` / `Viewport`        | interaction, architecture | A camera has `(x, y, z)` (z = zoom). The viewport applies camera transforms to world coordinates. tldraw separates camera state from viewport DOM size and offers `setCamera`, `centerOnPoint`, `zoomToFit`. | Keiko's `View { zoom, x, y }` plus `panBy` and zoom helpers are functionally identical. Fit-to-view is implementable on top of the current surface.                                                             | adapt concept (existing surface) ŌĆö record `View` as the camera contract; add fit-to-view helper inside `useWorkspace` only if a downstream issue actually needs it |
| `Shape` / `ShapeUtil`        | architecture              | A shape is a typed record `{ id, type, x, y, props }`; a `ShapeUtil` declares behavior (render, hit, resize, indicator, geometry). New shape types register via `defineShape`.                               | Keiko's `AppWindow { id, type, x, y, w, h, z, cfg }` mirrors the shape record; `WindowTypeDef` plus the `WIN_META` sidecar table mirrors the descriptor contract; `registerWindowRender` mirrors `defineShape`. | adopt concept ŌĆö confirm the parallel in #525 ADR; land governance metadata through `WorkspaceDescriptorMeta` + `WIN_META`, not a second registry                   |
| `Tool`                       | interaction               | Tool state machine for the active interaction mode (select, draw, hand, etc.). Tools own pointer state, cursor, and gestures.                                                                                | Keiko's interactions are command-driven (palette + buttons) rather than mode-driven. The command palette + window-level interactions cover the same need without a state-machine.                               | adapt concept (rejected as a state machine, adopted as commands) ŌĆö record in #523 UX blueprint                                                                     |
| `Store` / `Records`          | architecture              | Reactive document store of typed records with serializable schema. tldraw uses it for persistence and sync.                                                                                                  | Keiko's UI state lives in React hooks (`useWorkspace`, contexts). The current shell persists layout to browser `localStorage`; evidence and workspace FS have their own stores.                                 | adapt concept (rejected as a single store) ŌĆö keep multi-store ownership per existing package boundaries                                                            |
| Undo/redo                    | interaction               | tldraw exposes a `history` API and command pattern. Most editor actions are undoable; programmatic actions are explicitly excluded.                                                                          | Not implemented today. Per audit Gap Matrix, a typed undo stack lands in #527, scoped to UI-state actions and refusing to mutate evidence / patches / verification / model-call records.                        | adapt concept ŌĆö fully specified in #523 UX blueprint and #525 ADR; implemented in #527                                                                             |
| Collaboration boundary       | architecture              | tldraw v3 added multiplayer via `useSyncStore` and a sync server.                                                                                                                                            | Real-time collaboration is explicitly out of scope for #518. WebSocket already in use for streaming; WebRTC is deferred.                                                                                        | reject (explicit non-goal)                                                                                                                                         |
| Virtualization / performance | architecture              | tldraw culls off-screen shapes, batches paints, and uses `requestAnimationFrame` for camera animation.                                                                                                       | Keiko's window count is small (Ōēżdozens); culling is unnecessary at present scale. Existing `useWorkspace` already minimizes re-renders by memoizing window arrays.                                              | defer ŌĆö record as future ADR candidate if window counts grow                                                                                                       |
| React/TypeScript integration | architecture              | tldraw exposes typed hooks (`useEditor`, `useValue`, `useUiContext`).                                                                                                                                        | Keiko's `useWorkspace`/`useChatSession`/`useTwinMode`/`useTheme` hooks already form a typed integration layer.                                                                                                  | adapt concept (existing surface)                                                                                                                                   |

### Concepts explicitly rejected

- **`@tldraw/tldraw` runtime dependency.** Forbidden by the no-new-dependency invariant. Substrate already exists in Keiko.
- **tldraw `defineMigrations` schema-versioning system.** Keiko already has `evidenceSchemaVersion` and `node:sqlite` `user_version` patterns; no second migration system is needed.
- **tldraw "asset" subsystem.** Out of scope for #518.

| --------------------------------- | -------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| Document + whiteboard composition | product, interaction | AFFiNE composes a document mode and an edgeless (whiteboard) mode that share a block model.    | Keiko already composes a desktop workspace with multiple object types (chat, files, editor, browser, terminal, review, agents, integ, project, plus panels). The "document/whiteboard" duality already exists as `WindowType` plus `tool` panels. | adapt concept (existing surface)                                                                           |

Existing `SafeMarkdown` renders rich text; existing widgets render typed cards. The unit of composition is the window/widget, not the block.                         | reject (out of scope) ŌĆö re-evaluate if a future "Document object" issue requires it                        |

| Knowledge base / linked notes     | product              | AFFiNE exposes bidirectional links and a knowledge graph.                                      | Keiko's local-knowledge package already implements a capsule graph with a connector graph UI.                                                                                                                                                     | adapt concept (existing surface)                                                                           |
| Persistence / sync boundary       | architecture         | AFFiNE separates local store from optional Cloud sync.                                         | Keiko has no Cloud sync; evidence + workspace + memory stores are local.                                                                                                                                                                          | reject (explicit non-goal) ŌĆö record local-only constraint in #525 ADR                                      |
| Multi-workspace switching         | product, interaction | AFFiNE supports multiple top-level workspaces.                                                 | Keiko already has a project-oriented sidebar concept; `ProjectPanel` consumes live project/chat state via `useChatSessionContext` and is not a full multi-workspace switcher (it does not expose a registered-project list).                      | adapt concept ŌĆö extend the existing sidebar surface when a future issue adds a registered-project switcher |
| Knowledge-work workflows          | product              | AFFiNE has templates, AI commands, and structured property pages.                              | Keiko has workflow descriptors, the run/evidence ledger, and grounded chat.                                                                                                                                                                       | adapt concept (existing surface)                                                                           |

### Concepts explicitly rejected

- **`@blocksuite/*` runtime dependencies.** Forbidden by the no-new-dependency invariant.
- **AFFiNE Cloud sync.** Out of scope; Keiko is local-first.
- **AFFiNE plugin host.** Reuse `registerWindowRender` if extension is needed; no runtime plugin host beyond that.

## Reference project 3 ŌĆö Excalidraw

### Product, interaction, and architecture concepts covered

| Excalidraw concept | Concept class             | Public behavior summary                                                                                           | Keiko mapping                                                                                                                                                                                              | Disposition                                                                               |
| ------------------ | ------------------------- | ----------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Scene model        | architecture              | A flat array of `ExcalidrawElement` records; rendering iterates the array.                                        | Keiko's `AppWindow[]` is functionally identical.                                                                                                                                                           | adapt concept (existing surface)                                                          |
| Coordinate system  | interaction, architecture | World coordinates with a `(scrollX, scrollY, zoom)` viewport. Pointer events are converted via affine transforms. | Keiko's `View { zoom, x, y }` plus `Workspace.tsx` pointer handlers already implement this.                                                                                                                | adapt concept (existing surface)                                                          |
| Renderer           | architecture              | Excalidraw renders to a 2D canvas. Each frame redraws affected layers.                                            | Keiko renders windows as DOM elements with CSS transforms (the workspace background is a `WorkspaceShader` div). DOM rendering is sufficient at Keiko's element count and provides accessibility for free. | adapt concept (rejected as canvas) ŌĆö explicit DOM rendering decision recorded in #525 ADR |
| Hit detection      | interaction               | Excalidraw uses bounding-box hit tests plus geometric intersection helpers.                                       | Keiko uses DOM hit-testing for windows (browser handles it) and explicit hit logic in `connectionUtils.ts` for connections.                                                                                | adapt concept (existing surface)                                                          |
| Selection logic    | interaction               | Excalidraw supports single, multi, and marquee selection.                                                         | Keiko's selection is single-window today; multi-selection is a candidate extension for #527 if #523 UX requires it.                                                                                        | adapt concept (extension in #527)                                                         |
| Command system     | interaction, architecture | Excalidraw commands are dispatched as actions through a reducer.                                                  | Keiko commands are typed records routed through `CommandPalette` and the `WorkspaceApi`.                                                                                                                   | adapt concept (existing surface) ŌĆö no reducer is needed                                   |
| Collaboration      | architecture              | Excalidraw supports collaboration via Firebase.                                                                   | Out of scope for #518.                                                                                                                                                                                     | reject                                                                                    |

### Concepts explicitly rejected

- **`@excalidraw/excalidraw` runtime dependency.** Forbidden by the no-new-dependency invariant.
- **Roughjs / hand-drawn render style.** Not a Keiko design direction.
- **`React-Konva` or `canvas` library adoption.** DOM rendering is sufficient.

## No-new-dependency compliance note

Every adopted/adapted concept in this analysis is implementable on Keiko's existing stack and platform APIs:

| Capability            | Implementation seam in Keiko                                                                                                       | New dependency?                 |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| Editor pattern        | React hooks (`useWorkspace`) + TypeScript types                                                                                    | No                              |
| Camera transforms     | Pure math (`zoom`, `x`, `y`) + CSS `transform` on the workspace surface                                                            | No                              |
| Shape/object registry | TypeScript discriminated union + a single registry map                                                                             | No                              |
| Undo stack            | Pure data structure with typed `Action` records                                                                                    | No                              |
| DOM rendering         | React component tree + CSS                                                                                                         | No                              |
| Node + edge rendering | Existing SVG/CSS patterns in `ConnectionsLayer` and `connector-graph`                                                              | No                              |
| Persistence           | Browser `localStorage` for the current workspace shell layout; existing local package stores for evidence and other owned surfaces | No                              |
| Pointer behavior      | `pointerdown`/`pointermove`/`pointerup` + `PointerEvent` (browser native)                                                          | No                              |
| Keyboard handling     | `keydown` listeners + Mousetrap-style keymap object (pure data)                                                                    | No                              |
| Accessibility         | Existing `jest-axe`, axe-core, aria patterns, focus-visible                                                                        | No                              |
| WebSocket streaming   | Existing `ws` already in the product architecture                                                                                  | No                              |
| Future WebRTC         | Not implemented; if a future ADR approves it, browser native `RTCPeerConnection` is used                                           | No (browser native if approved) |

The capability audit's Gap Matrix produces zero new runtime dependencies. This reference analysis confirms that constraint at the concept level: no concept selected for adoption requires a third-party library.

## Keiko workspace glossary (reference-mapped)

| Term                  | Keiko meaning                                                                                                                                                                 | Reference origin                               | Existing Keiko surface                                                        |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------- | ----------------------------------------------------------------------------- |
| **World coordinates** | The continuous 2D plane on which workspace objects are placed.                                                                                                                | tldraw, Excalidraw                             | `Workspace.tsx` view transforms                                               |
| **Camera**            | The user's current view of the world: `{ zoom, x, y }`.                                                                                                                       | tldraw                                         | `View` type in `windows/types.ts`; `useWorkspace` zoom/pan                    |
| **Viewport**          | The DOM rectangle through which the camera projects the world.                                                                                                                | tldraw, Excalidraw                             | `Workspace.tsx` container div                                                 |
| **Renderer**          | The mechanism that draws workspace objects. Keiko's renderer is DOM-based React, not a canvas.                                                                                | tldraw, Excalidraw                             | `Workspace.tsx` + window components                                           |
| **Object registry**   | The typed catalog of workspace object types.                                                                                                                                  | tldraw `ShapeUtil`, React Flow `nodeTypes`     | `WindowsRegistry.ts`                                                          |
| **Object descriptor** | The typed declaration for a single object type: title, icon, size, config, render, lifecycle, trust boundary, authority, persistence.                                         | tldraw `ShapeUtil`, React Flow `nodeTypes`     | `WindowTypeDef` + `WIN_META`                                                  |
| **Object record**     | A live instance of an object in the workspace: id, type, position, size, z-index, config.                                                                                     | tldraw `Shape`, Excalidraw `ExcalidrawElement` | `AppWindow`                                                                   |
| **Command**           | A discoverable, typed user action routed through the command palette and the workspace API.                                                                                   | tldraw `Editor` commands, Excalidraw actions   | `Command` in `CommandPalette.tsx` + `buildAppShellCommands` in `AppShell.tsx` |
| **Tool**              | A high-level interaction mode. Keiko prefers commands over modal tools; "tool" in Keiko refers to a workspace window classified as a `tool` panel in the registry.            | tldraw                                         | `WindowTypeDef.tool` flag                                                     |
| **Selection**         | The set of workspace objects currently the focus of commands. Single today; multi-selection lands in #527 if #523 requires it.                                                | All four                                       | `useWorkspace` focus + #527 extension                                         |
| **Connection**        | A persisted relationship between two workspace objects, rendered as a line.                                                                                                   | React Flow edges                               | `Connection` in `windows/types.ts` + `ConnectionsLayer`                       |
| **Node**              | A capsule, project, or future graph entity rendered in the connector graph. Distinct from a workspace window.                                                                 | React Flow                                     | `CapsuleListEntry` in `connector-graph-types.ts`                              |
| **Edge**              | A typed relationship between nodes.                                                                                                                                           | React Flow                                     | `Connection` type in `windows/types.ts` + `ConnectionsLayer`                  |
| **Panel**             | A workspace window classified as a sidecar tool (project, search, plugins, inspector, etc.).                                                                                  | AFFiNE                                         | `WindowTypeDef.tool === true`                                                 |
| **Inspector**         | The window or panel that shows metadata for a selected object.                                                                                                                | AFFiNE, tldraw                                 | `InspectorPanel.tsx`                                                          |
| **Evidence object**   | A workspace-visible reference to an entry in the evidence ledger. The descriptor records that the persisted state is an evidence reference, not the evidence itself.          | n/a                                            | `keiko-evidence` + UI evidence widgets                                        |
| **Authority moment**  | A point where the user explicitly consents to apply, verify, dismiss, archive, or escalate.                                                                                   | n/a (Keiko-original)                           | `PermControl`, `AgentGateCard`, `ReviewWidget`                                |
| **Trust boundary**    | The package/module boundary enforced by Keiko architecture (Model Gateway, Workspace, Tools, Evidence). Workspace objects declare which trust boundaries their actions cross. | n/a                                            | ADR-0019, ADR-0020                                                            |

## Follow-up ADR candidates

The analysis named the following epic-linked ADR follow-ups for [issue #525](https://github.com/oscharko-dev/Keiko/issues/525). They now exist on `dev` and remain the authoritative disposition for the concepts mapped in this document:

1. [ADR-0026 ŌĆö Workspace substrate](../adr/ADR-0026-workspace-substrate.md): `useWorkspace` is the editor; DOM rendering is the renderer; `View` is the camera and `Workspace.tsx` is the viewport.
2. [ADR-0029 ŌĆö Workspace object registry and extension contract](../adr/ADR-0029-workspace-object-registry.md): the registry sidecar defines `lifecycle`, `trustBoundary`, `authority`, and `persistence`.
3. [ADR-0028 ŌĆö Workspace commands, events, selection, undo/redo boundaries](../adr/ADR-0028-workspace-commands-undo.md): reversible UI actions are typed; evidence, patches, verification, and model calls are refusal-by-type.
4. [ADR-0027 ŌĆö Workspace state ownership and persistence boundaries](../adr/ADR-0027-workspace-state-ownership.md): per-package stores remain explicit and no new persistence store is introduced.
5. [ADR-0030 ŌĆö Workspace security, evidence, and trust boundaries](../adr/ADR-0030-workspace-security-evidence.md): model calls stay behind the Model Gateway, tool execution stays behind terminal policy, and evidence remains redacted by construction.
