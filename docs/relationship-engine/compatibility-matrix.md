# Epic #532 — Relationship Compatibility Matrix

Status: Wave 2 deliverable for [issue #534](https://github.com/oscharko-dev/Keiko/issues/534) under parent epic [#532](https://github.com/oscharko-dev/Keiko/issues/532). Companion to [taxonomy.md](taxonomy.md), [denial-reasons.md](denial-reasons.md), [lifecycle.md](lifecycle.md).

Issue date: 2026-06-06.

## Purpose

This document tabulates the source-kind × target-kind compatibility for every relationship type defined in [taxonomy.md §5](taxonomy.md). It is the binding input for the deterministic validator in issue [#538](https://github.com/oscharko-dev/Keiko/issues/538). Cell values match the relationship type ids and denial reason codes in the taxonomy.

The matrix is normative. Any cell that is not explicitly listed as allowed is **denied**. The deterministic validator MUST refuse any pair not listed in §2 with the matching denial reason code in §3.

## 3. Explicit denial examples

The following cells are called out so the validator's negative tests and the UI inspector's error rendering can pin them.

| Source           | Target           | Outcome                                   | Rationale                                                                                                                                                                                                                              |
| ---------------- | ---------------- | ----------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `chat`           | `chat`           | `denied/source-kind-not-allowed`          | Chats do not relate to chats in the relationship engine. Cross-chat coupling, if needed, goes through a shared `workflow-run` or a shared `capsule`. See [taxonomy.md §8](taxonomy.md).                                                |
| `workspace-path` | `workspace-path` | `denied/source-kind-not-allowed`          | Path-to-path edges live in `importGraph` ([`packages/keiko-workspace/src/importGraph.ts`](../../packages/keiko-workspace/src/importGraph.ts)), not in the cross-domain relationship engine.                                            |
| `capsule`        | `capsule`        | `denied/source-kind-not-allowed`          | Capsule-to-capsule edges within a capsule-set are modelled by the local-knowledge connector graph ([`packages/keiko-contracts/src/local-knowledge.ts:204`](../../packages/keiko-contracts/src/local-knowledge.ts)).                    |
| `agent`          | `evidence-run`   | `denied/kind-incompatible` (post-landing) | Evidence is produced by runs, not by agents. The lineage is agent → run → evidence, not agent → evidence. Validator returns `denied/kind-incompatible` once the `agent` kind lands; until then `denied/object-kind-not-yet-supported`. |
| `evidence-run`   | (any)            | `denied/source-kind-not-allowed`          | Evidence runs are leaf artefacts; they do not initiate edges.                                                                                                                                                                          |
| `tool`           | (any)            | `denied/source-kind-not-allowed`          | A tool registry entry does not originate relationships; runs do (`uses-tool` is `workflow-run` → `tool`).                                                                                                                              |
| `patch-proposal` | (any)            | `denied/source-kind-not-allowed`          | A patch proposal is a target of `proposes-patch`, never a source. Apply / reject is recorded by the harness event, not by reverse edges.                                                                                               |
| `workspace-path` | (any)            | `denied/source-kind-not-allowed`          | Workspace paths are passive endpoints. Edges that touch a path are originated by a `workflow-run` or a `chat`.                                                                                                                         |
| `workflow-run`   | `chat`           | `denied/target-kind-not-allowed`          | Runs do not point back at the originating chat as a target; the inverse `chat → workflow-run` via `starts-workflow` already records the lineage.                                                                                       |

## 4. Allowed-pair examples (for downstream test fixtures)

These rows enumerate at least one positive case per relationship type so the deterministic validator (#538) has a ground-truth set.

| Relationship type     | Source kind    | Target kind      | Notes                                                                                               |
| --------------------- | -------------- | ---------------- | --------------------------------------------------------------------------------------------------- |
| `reads-context`       | `workflow-run` | `memory`         | The canonical "the run retrieved this memory" edge. Anchored to `memory:retrieved` audit event.     |
| `reads-context`       | `workflow-run` | `capsule`        | The canonical "the run read this capsule" edge.                                                     |
| `reads-context`       | `workflow-run` | `workspace-path` | The canonical "the run read this file" edge. `assertContainedRealPath` gates resolution.            |
| `reads-context`       | `chat`         | `evidence-run`   | A chat surfaces an earlier evidence run as context.                                                 |
| `proposes-patch`      | `workflow-run` | `workspace-path` | The canonical patch-proposal-touching-a-file edge.                                                  |
| `proposes-patch`      | `workflow-run` | `patch-proposal` | Pinning the proposal by harness event id.                                                           |
| `uses-tool`           | `workflow-run` | `tool`           | The canonical "the run used this tool" edge.                                                        |
| `starts-workflow`     | `chat`         | `workflow-run`   | The canonical "a user message started a run" edge.                                                  |
| `starts-workflow`     | `workflow-run` | `workflow-run`   | A parent run spawns a child run.                                                                    |
| `produces-evidence`   | `workflow-run` | `evidence-run`   | The 1:1 lineage edge from a run to its persisted manifest.                                          |
| `references-document` | `chat`         | `workspace-path` | A chat is grounded in a file.                                                                       |
| `references-document` | `chat`         | `capsule`        | A chat is grounded in a capsule.                                                                    |
| `references-document` | `workflow-run` | `capsule-set`    | A run is grounded in a capsule set.                                                                 |
| `depends-on`          | `memory`       | `memory`         | The cross-domain reification of `MemoryEdgeKind "derived-from"` semantic at the relationship layer. |
| `depends-on`          | `workflow-run` | `evidence-run`   | A run depends on a prior evidence run as substrate.                                                 |
| `depends-on`          | `capsule-set`  | `capsule`        | A set depends on a member capsule.                                                                  |
| `depends-on`          | `memory`       | `workspace-path` | A memory's provenance depends on a workspace path.                                                  |
